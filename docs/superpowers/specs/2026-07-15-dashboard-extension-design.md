# Dashboard extension: Product + Margin, Customer analytics

**Date:** 2026-07-15 · **Status:** approved · **Scope decided by user:** Product+Margin ✓, Customer ✓
(Delivery ops & Payment/Promo deferred). Model built via API; report visuals assembled
manually from the layout spec below (PBIR edits ruled out as fragile).

## Data layer — `nb_bi_refresh` only (pipeline core untouched)

`vw_eod_sales` gains columns already present in the gold fact:
- `customer_id`, `customer_gender`, `customer_age_range`, `customer_nationality` (NULLIF '' → cleaner slicers), `rating`
- `cogs = purchase_price_without_tax * quantity` (unit cost from the effective-dated price timeline)

New mart `gold.bi_dim_product` (Direct Lake-safe current view of the SCD2 dim):
`silver.dim_product WHERE is_current` ⟕ `silver.dim_oms_product` →
product_id, product_name, product_category, product_group, product_sub_category,
retail_business_type, brand, manufacturer.

`bi_eod_sales` (`SELECT * FROM vw_eod_sales`) picks the new columns up automatically.

## Semantic model (TMDL via updateDefinition)

- Sales: +6 columns (customer_id int64, cogs double, rating int64, 3 demographic strings)
- New table `dim_product` ← `[gold].[bi_dim_product]` (directLake partition, same expressionSource)
- Relationships: `Sales.product_id → dim_product.product_id`;
  `'Sales Daily'.report_date → dim_date.date` (fixes the orphaned aggregate table)
- Measures on Sales: COGS, Gross Margin, Margin %, Unique Customers, Identified Baskets,
  Identified Rate %, Repeat Customers (≥2 baskets in filter context), Repeat Rate %, Avg Rating

## Validation

After materialize + reframe: [Revenue] total must equal the pre-change total (existing
columns untouched); Margin % in a sane band; dim_product slicing shows no dominant
blank bucket; existing report unaffected (no PBIR change).

## Known caveats (stated to user)

- `dim_product` slices by CURRENT attributes (is_current); the fact's own category
  columns are transaction-time snapshots — they can differ after a product recategorizes.
  Layout spec uses dim_product for slicers.
- `rating` only exists on delivery transactions → Avg Rating is delivery-scoped.
- `vw_eod_sales` (unchanged behavior) excludes refunds/canceled/test stores.

## Implementation notes (discovered during rollout)

1. A measure may not share a name (case-insensitive) with a column in its table →
   measure is `Total COGS`, column is `cogs`.
2. The Direct Lake-on-OneLake expression had `PBI_IncludeFutureArtifacts = False`,
   which blocks framing of tables created AFTER the model was bound (`bi_dim_product`
   → "source table does not exist"). Flipped to `True`.
3. `Unique Customers` / `Repeat Customers` filter `customer_id > 0` — gold uses
   `coalesce(user_id, 0)`, so 0 is the anonymous placeholder, not a customer.
4. `executeQueries` accepts exactly one DAX query per request.

Validation results: [Revenue] = baseline ± 3.48 VND (daily-mart rounding);
Margin % = 28.94% (consistent with simulator cost = 0.7 × price, less promos);
Unique Customers = 30 (= N_USERS exactly); category split shows no blank bucket.

## Update: report WAS created via API after all (user requested the experiment)

Report **"EOD Sales - Product & Customer"** (`58f8e953-6aaa-410b-beb6-ac615c3b3d2d`),
PBIR enhanced format, generated by `sample_file/create_report_pbir.py`, bound to the
same model; the existing dashboard was not touched. Verified by exporting to PDF via
the Power BI export API and inspecting the rendered pages — all 19 visuals render with
live data. One bug found & fixed: setting `"format": "G"` on card projections overrides
the measure's formatString (Margin % showed 0.2893…); removing the projection-level
format lets the measure format apply (28.9%). PBIR learnings: visual role names are
Category/Y/Tooltips (bar), Group/Details/Values (treemap), Rows/Columns/Values
(pivotTable), Category/X/Y/Size (scatter), Data (card), Values (slicer); a report is
importable without filterConfig; themes + report.json can be reused verbatim.
